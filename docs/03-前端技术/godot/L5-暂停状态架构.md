# L5 暂停状态架构

> 本文锁定 Godot 全局暂停与游戏状态机的边界。  
> 版本归属：V0.4 | 生命周期状态：已审核

## 设计结论

`GameController` 是暂停业务状态的唯一归属：它将当前 `READY`、`BALLS_ACTIVE` 或 `RESOLVING` 保存为 `_state_before_pause`，切换到 `PAUSED` 后拒绝发射与后续业务推进。`Main` 负责将此业务操作映射为 `SceneTree.paused`，从而一次性冻结默认处理模式下的球物理、计时器、棋盘 Tween 和挑战时钟。

`PauseMenu` 是唯一使用 `PROCESS_MODE_ALWAYS` 的局内暂停 UI；它展示遮罩并发出 `resume_requested`，但不直接修改游戏规则或 `SceneTree`，避免 UI 与玩法控制器耦合。

重新开始也只能由暂停菜单触发：`Main` 先解除 `SceneTree.paused`，再调用 `GameController.restart_game()`；控制器停止挑战时钟、清空 `BallManager` 活动序列和 `BoardController` 的节点/逻辑格/Tween，最后以已持有的模式 Resource 执行同一套新局初始化。

## 状态流转

```text
READY / BALLS_ACTIVE / RESOLVING
  → GameController.pause_game()
  → PAUSED + 记录原状态
  → Main 设置 SceneTree.paused = true，并显示 PauseMenu

PauseMenu.resume_requested
  → Main 取消 SceneTree.paused
  → GameController.resume_game()
  → 恢复记录的原状态
```

## 约束

- 先切换业务状态和显示菜单，再冻结树；恢复时先解除树冻结和隐藏菜单，再恢复业务状态，保证暂停菜单始终可交互。
- `GameController._process()`、`_physics_process()` 与 `can_request_launch()` 都必须显式识别 `PAUSED`，以防未来有节点采用 `PROCESS_MODE_ALWAYS` 时绕过 `SceneTree.paused`。
- 任何后续新增的局内计时器都应保持默认可暂停处理模式；若必须 `ALWAYS` 运行，则必须由 `PAUSED` 状态显式阻断。
