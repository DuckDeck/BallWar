# L5 架构规范：Godot 客户端

> 所属功能域：Godot 客户端架构  
> 版本归属：V0.1-MVP | 生命周期状态：已审核

## 架构骨架

```text
Main
├── ArenaRenderer       场地程序绘制
├── Boundaries          物理边界
├── ObstacleLayer       障碍实例容器
├── BallLayer           球实例容器
├── Launcher            输入与瞄准绘制
├── BallManager         顺序发射、活动球与批次边界
├── BoardController     网格真值、波次推进与危险线
├── GameController      局内状态、得分与回合协调
└── CanvasLayer         临时 HUD
```

| 层 | 职责 | 不允许承担的职责 |
|---|---|---|
| 场景与图层 | 组合可复用节点，提供局部容器 | 承担玩法状态或全局单例职责 |
| `GameController` | 局内状态、计分、批次完成后的回合收束 | 绘制、读取原始拖拽细节、逐球创建或直接操纵碰撞形状 |
| `BallManager` | 顺序创建/注销球、维护待发与活动球集合、发布批次完成 | 方块受击、得分、棋盘推进 |
| `BoardController` | 逻辑网格、方块投影、底行生成、危险线与一次性波次推进 | 球的物理或发射队列 |
| `Ball` / `Obstacle` | 自身物理/回收或生命值，并以 signal 通知结果 | 查找远处 UI 或修改全局状态 |
| `Launcher` | 鼠标/触摸输入和瞄准视觉 | 创建球、计分或决定回合状态 |
| `ArenaRenderer` / HUD | 视觉呈现 | 玩法判断和物理计算 |

## 目录与命名

- `scenes/`：`snake_case.tscn`，根节点职责单一。
- `scripts/`：文件和 `class_name` 一一对应；类用 PascalCase，函数、变量、signal 用 snake_case。
- `resources/`：共享参数的 `.tres`；关键手感参数不得在多脚本重复硬编码。
- 所有变量、参数、返回值、typed signal 必须显式声明类型。

## 技术选型与禁止模式

- 使用 Godot 4.7、GDScript、`CharacterBody2D` 的显式速度和 `StaticBody2D` 的障碍碰撞；共享参数采用 `GameConfig` Resource。
- 局内状态不使用 Autoload；只有未来出现真正跨场景的系统时才可经设计评审引入。
- 禁止 `yield()`、字符串形式 signal 连接、`PackedScene.instance()`、未类型化变量、在 `_process` 中查找节点、跨层直接改写 UI。
- 鼠标/触摸通过输入节点处理；未来 Android TV 输入必须映射到项目 Input Map，且仅在 T6 的 Android 验证之后加入。

## 当前扩展性评估

当前结构支持无人工上限的顺序发射库存、独立拖尾与颜色、六种单格障碍、按批次结算的波次推进。库存很高时，顶部预览的局部碰撞计算是优先性能观测点；`BallDefinition → BallEffect → HitContext/HitResult` 已是特殊球的正确扩展入口；`BoardState → BoardLayout → BoardController` 也已把方块逻辑格与节点视图分开。

以下是下一阶段开始实现爆炸、穿透、加大体积或奖励球前的优化顺序；它们是技术建议，不改变当前已拍板玩法：

1. **P0：以 `GameMode`、球库存和波次策略取代单一回合成长。** `RoundPlan` 只冻结一次发射的 `BallDefinition`；普通/挑战模式的波次触发分别由批次与独立时钟决定，球数最终由奖励块更新的库存提供。这样临时加球、特殊球库存和回放不会再依赖“安全回合数”。
2. **P0：把回收动画从 `Ball` 中抽为独立组件。** 保留 `Ball` 负责碰撞、速度和命中编排，将 `ROLLING → LIFTING → ROOF_RETURN` 的路径推进交给 `BallRecoveryPath`，把拖尾绘制交给子节点/组件。当前 `Ball` 仍可维护；但在增加四种以上特殊球效果前应完成这次拆分，避免一个节点同时承担物理、特效和路径状态。
3. **P1：将障碍形状由枚举分支提升为 `ObstacleDefinition` Resource。** Resource 应保存几何点、碰撞形状工厂、绘制风格、旋转策略与生成权重。现有六种基础形状继续可用；新形状或带行为的障碍无需同时修改 `WaveGenerator` 和 `Obstacle` 的多处 `match`。
4. **P1：为范围效果加入棋盘命中服务/队列。** 爆炸、穿透等效果通过命中队列请求 `BoardController` 对多个目标执行 `receive_hit`，统一在本物理步之后提交销毁与得分。避免效果在单次碰撞回调内遍历并释放多个障碍，造成重复得分或失效节点访问。
5. **P2：保持随机流可拆分。** 目前列、生命值、形状和旋转共用一个带种子随机流；新增任一随机字段会改变同一种子的后续布局。需要严格复现或录像时，拆为布局/属性两个随机流，或把生成的 `WaveEntry` 完整记录下来。

## 运行期护栏

- `force_recover()` 只应被测试调用；正式玩法不得以它跳过实际回收路径。后续可重命名为 `force_complete_for_test()` 或放入测试适配层，防止重新引入“球还在回程就开始下一回合”的缺陷。
- `ball_max_lifetime` 当前语义是“强制进入受控回收”，不是立即删除球。后续重命名为 `ball_forced_recovery_seconds` 会更贴合行为，配置 UI/文档也不易误用。
- 10 球基线无需对象池；如果目标提升到 30 颗以上同时活动球，应先为 `Ball` 和拖尾的 `Line2D` 增加池化与点数上限，再增加视觉密度。
