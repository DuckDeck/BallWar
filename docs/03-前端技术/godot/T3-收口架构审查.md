# T3 收口架构审查

> 项目形态：离线 2D 客户端  
> 版本归属：V0.3 | 生命周期状态：草稿

## 结论

T3 的棋盘、波次、重球和奖励格主链路职责清晰，未发现会阻断当前经典/挑战玩法的 P0 问题。`GameController` 负责模式和库存编排，`BoardController` 保持逻辑格真值，`WaveGenerator` 只生成行数据，`BallManager` 保持活动批次归属；这套边界足以支撑当前两种奖励和重球。

## 优化清单

| 优先级 | 项目 | 观察 | 建议时机 |
|---|---|---|---|
| P1 | 重球随机种子 | `heavy_ball_random_seed` 为非零固定值，每次新局的重球分布会重复；这与正式对局的波次随机策略不一致。 | 下次调重球出现率时，采用与 `wave_seed` 相同的 `0 = 每局随机、非零 = 调试复现` 约定，并补充分布回归。 |
| P1 | 重开复位 | `GameController.start_game()` 目前只允许从模式选择进入；`BoardController.initialize_board()` 不清空既有节点和逻辑格。T4 加入“再来一局”后若直接复用场景，会遗留棋盘、批次 ID 或滚屏 Tween。 | T4 实现重开入口前，补一个显式 `reset_game()`/`reset_board()`，停止时钟与 Tween、释放棋盘节点、重置 BoardState 和 Controller 库存。 |
| 已解决 | 人工球数上限 | 旧版 `maximum_ball_count` 会静默吞掉 `+1` 或重球复制奖励。 | 已移除上限与对应拦截；库存球持续累加，后续只按性能需求优化预览模拟。 |
| P2 | 棋盘命名 | `BoardState` 的公开方法与映射仍使用 `obstacle` 命名，但其中已登记普通障碍和奖励块。 | 下次新增第三种棋盘内容时，统一迁移为 `board_node`/`board_item`，减少新功能的语义负担。 |
| P2 | 配置清理 | `GameConfig.initial_obstacle_position` 已没有运行时引用。 | T4 前的配置整理中删除，或明确标记为仅供旧测试场景使用。 |
| P2 | 奖励端到端覆盖 | 奖励专项覆盖了 `+1` 的真实收集；放大/重球复制路径部分通过直接调用 Controller 验证。 | T5 前补一条真实碰撞的放大奖励端到端测试，并断言碰撞体、拖尾和库存的最终状态。 |
| P3 | 对象池 | 目前奖励/球按场景实例化和释放，当前上限 10 与低波次频率下足够稳定。 | 只有将最大活动球数或特效数量显著提高后，再引入球/反馈对象池。 |

## 保持不改的边界

- 不将奖励掉落球放进待发预览；它必须属于命中球的活动批次，才能避免提前结算。
- 不让 `WaveGenerator` 生成节点或修改分数；它只生成可复现的行数据。
- 不让 `BoardController` 直接决定球库存；奖励效果仍由 `GameController` 编排。
- 不为当前十球以内的场景过早引入 Autoload 或对象池。
