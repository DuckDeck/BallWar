# L5 实体扩展规范：球、障碍与波次

> 本文定义后续玩法的实现边界；具体玩法规则必须先由 L1 需求拍板，本文不替代产品决策。  
> 版本归属：V0.2（建议） | 生命周期状态：草案

## 适用范围

当需求涉及多球、波次下压、爆炸/穿透/大体积球、更多形状或行为的障碍时，先补齐对应 L1/L2，再按本文拆分代码。第一阶段普通球和六边形块可作为默认定义迁移，不应重写已验证的基础手感。

## 职责边界

| 模块 | 应负责 | 不应负责 |
|---|---|---|
| `BallDefinition` Resource | 半径、初速、重力倍率、外观、拖尾、能力列表等静态定义。 | 本局生命、命中过的对象、当前速度。 |
| `Ball` | 移动、碰撞探测、生命周期、实例状态；同步物理碰撞半径。 | 识别具体障碍类型、计算爆炸范围、决定回合结束。 |
| `BallEffect` | 特殊球在发射、命中、回收等时点附加的行为。 | 直接修改 HUD 或 Controller 私有状态。 |
| `HitResolver` | 根据命中上下文结算伤害、反弹、穿透、范围效果和去重。 | 绘制球或维护关卡生成。 |
| `ObstacleDefinition` Resource | 形状、生命、得分、反弹材质、掉落/行为配置。 | 运行时生命与节点引用。 |
| `Obstacle`/行为组件 | 统一受击、显示、碰撞和该块特有运动。 | 读取某种球的具体 class 后写分支。 |
| `BallManager` | 注册/注销活动球；所有球结束后通知回合结算。 | 生成波次和定义障碍内容。 |
| `LevelSpawner` | 根据波次/关卡数据生成多个障碍。 | 处理每帧球物理。 |
| `ArenaGeometry`/`BallRecoveryPath` | 场地几何、托底与回顶路线。 | 存储每颗球的能力状态。 |
| `GameController` | 回合、分数、暂停后的编排。 | 保存单球/单障碍假设或类型 switch。 |

## 命中契约

实现前建立强类型 `HitContext`（`RefCounted` 或 Resource），至少包含：来源球、伤害、命中点、法线、入射速度、能力标签和本次命中 ID。可受击对象提供单一 `receive_hit(context)` 入口并报告结果。

普通球只是默认策略：造成 1 点伤害并按目标 `BounceProfile` 反弹。穿透、爆炸、弹簧等差异由上下文和策略处理，禁止继续扩大 `if collider is Obstacle` 或按类名的分支链。

## 数据与实例规则

- `.tres` 定义资源默认共享；运行时不得写回其字段。
- 每种球都必须同步视觉半径、`CircleShape2D.radius`、拖尾宽度、发射净空、托底高度和回收安全距离。
- 每种障碍的视觉轮廓与碰撞轮廓应来自同一形状数据或同一场景资产；禁止复制两套顶点后分别修改。
- 需要本局变体时，创建独立运行状态；带嵌套 Resource 的定义复制应使用 `duplicate_deep()`。

## 多球与回合规则

`GameController` 不再以单个 `_active_ball == null` 判定结束。由 `BallManager` 维护 `Array[Ball]` 或等价注册表；每颗球创建即注册、回收/销毁即注销，只有活动集合清空后才进入结算。分裂球、奖励球必须经过同一注册路径。

具体的同帧得分、爆炸连锁、球数量上限、波次下压和失败线属于玩法规则，必须在实现前由 L1 明确，不能由本规范隐式决定。

## 性能与测试门槛

- 高频生成的球、爆炸特效和命中浮字在规模稳定后使用对象池；回收对象关闭物理处理。
- 高速球使用有限次数的剩余位移/子步策略，避免单次碰撞遗漏连续目标。
- 新增每一种球或障碍，至少补：定义读取、命中结果、回合结算、极端边界四类自动化覆盖。
- Android TV 位于 Android 真机验证之后；输入统一经 Input Map，并验证方向、确认、返回和焦点可见性。
