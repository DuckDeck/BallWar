# L5 核心链路：基础弹球闭环

> 所属功能域：基础弹球闭环  
> 版本归属：V0.1-MVP | 生命周期状态：已审核

## 状态策略

局内状态仅由 `GameController` 维护：`READY → BALL_ACTIVE → RESOLVING → READY`。状态的目的不是展示 UI，而是保证一回合只有一颗活动球，并让回收后总能恢复可输入状态。

## 核心业务链路

| 链路 | 步骤与取舍 | 代码锚点 |
|---|---|---|
| 发射 | `ArenaRenderer` 与 `Boundaries` 使用同一组屋顶缺口坐标绘制和碰撞：两条斜顶在中央留出 96 像素开口，短竖线只收口两侧。`Launcher` 位于缺口中心，将其下方整屏作为输入区；READY 时绘制待发球，`Main` 订阅 `GameController.state_changed` 并在 BALL_ACTIVE/RESOLVING 隐藏它。点击或拖拽松开时，球从缺口向下创建。 | `ArenaRenderer`、`Launcher`、`Main`、`GameController` |
| 反弹、命中与拖尾 | 球的竖直速度在每个物理帧叠加 `GameConfig.ball_gravity`；碰撞后按边界或障碍法线反射。场地边界保留当前速度大小，障碍物则乘以 `GameConfig.obstacle_bounce_restitution`（当前 0.68）以削弱六边形反弹。`Ball` 内部的 `Line2D` 保存最近 `ball_trail_duration` 秒的全局坐标，以透明度和宽度渐变绘制收尖拖尾；拖尾不参与物理。向上反弹会自然减速和回落；障碍自身减少生命并发出事件。 | `Ball`、`Obstacle` |
| 清除与得分 | 障碍销毁事件由控制器统一累计 Score；普通命中不改变 Score。由控制器汇总保证 HUD 不与碰撞节点耦合。 | `GameController` |
| 回合收束 | 正常路径中，球触及托底曲线后依次进入 `ROLLING → LIFTING → ROOF_RETURN`，抵达 `launcher_position` 后才通知控制器；超时或越过 `recovery_y` 才走异常兜底。控制器短暂进入 RESOLVING，清理球、在需要时补充障碍并转回 READY。 | `Ball`、`GameController` |

## 参数策略

`GameConfig` `.tres` 是球速、重力、障碍物回弹系数、拖尾时长和宽度、寿命、场地尺寸、障碍生命与得分的唯一共享来源。实例只读取配置，不在运行时修改它；未来需要单局差异时必须复制 Resource，而不是污染共享资源。

## 跨平台输入策略

T1 的输入来自鼠标与单指事件，统一转换为场地坐标。T6 才会在 Android 真机流程通过后补充 Android TV 的方向键、确认和返回映射；本链路不假定桌面键鼠专属能力。

## 托底回收实现

`GameConfig.get_bottom_trough_y()` 是托底曲线的唯一高度来源；`ArenaRenderer` 与 `Ball` 都调用它，避免视觉和命中位置漂移。`Ball` 在普通 `ACTIVE` 物理态外增加 `ROLLING`、`LIFTING`、`ROOF_RETURN` 三个受控回收子状态：它们不执行重力和碰撞，而是依次沿底部曲线、侧墙外部、顶部斜线移动到 `launcher_position`。活动球抵达最后路径点才发出 `bottom_trough` 回收信号；`GameController` 无论障碍是否仍存在都必须由 `RESOLVING` 恢复到 `READY`。
