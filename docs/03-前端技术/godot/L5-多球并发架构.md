# L5 多球并发架构

> 本文落实多球的客户端实现约束；特殊球的具体效果仍等待后续 L1 决策。
> 版本归属：v0.3 | 生命周期状态：已审校

## 设计结论

多球由 `BallBatch`、`BallLaunchSequence`、`BallManager`、每球独立的 `BallRuntimeState` 与数据化的 `BallDefinition` 协作完成。`GameController` 只编排批次和回合，`BallManager` 管理发射队列与活动球，`Ball` 只推进自身物理和回收状态。

不使用 Autoload：活动球、障碍层和回合都属于当前主场景，退出场景时应一并释放。

## 当前实现锚点

- `BallManager` 按批次维护待发队列和活动球表，并保存该批的顶部回收终点：经典模式使用斜线前暂存点，挑战模式使用 `launcher_position`。球抵达所属终点后才发出 `recovered` 并从活动表注销；待发与活动均为空时才发出 `batch_finished(batch_id)`。奖励生成的掉落球使用命中源球的同一 `BallLaunchSequence` 注册，因此会计入活动集合并阻止该批提前完成。
- `BallLaunchSequence` 保存待发球定义、统一瞄准方向、发射原点和发射节拍；它不是场景节点，因此不会产生等待球碰撞。
- `Launcher` 接收待发定义并按模式绘制：两种模式都使用同一套顶部凹槽重力、斜坡与低反弹碰撞预览，最低球占据发射口、余球自然堆叠在上方。经典模式在回收时关闭凹槽、结算后才打开；挑战模式始终开放，当前顺序队列清空后可在其他球仍活动时再次发射。每次发射或顺序队列变更都由 `GameController` 发布最新预览定义，避免保留已发射球的陈旧画面。
- `BallDefinition.visual_color` 同时驱动球体和拖尾的 RGB；每个已发射球仍使用 Layer 2、Mask 1，只检测 WORLD。

## 场景职责

```text
Main
├── BallLayer
├── ObstacleLayer
├── BallManager
│   └── BallLaunchSequence（数据状态）
├── Launcher（顶部槽内的待发球预览）
└── GameController
    └── 创建 BallBatch，监听 batch_finished
```

| 模块 | 负责 | 不负责 |
|---|---|---|
| `GameController` | 创建批次、切换回合、接收批次结束、结算回合 | 逐帧遍历球、直接处理碰撞 |
| `BallManager` | 顺序创建/注册/注销球，维护待发与活动集合，发布队列视觉数据 | 伤害、得分、障碍类型判断 |
| `BallLaunchSequence` | 统一瞄准方向、待发定义、活动球表与发射节拍 | 物理移动和 HUD 绘制 |
| `Launcher` | 模式化的暂存/待发预览、瞄准输入与补位动画 | 创建物理球、回合结算 |
| `Ball` | 独立运动、WORLD 碰撞、同色拖尾、报告回收 | 检测或响应其他球 |

## 奖励球接入

- `BoardController` 将奖励格作为与障碍相同的逻辑占格节点生成，收集后先释放格位，再向 `GameController` 发出 `reward_collected`。
- `GameController` 处理规则：`+1` 创建普通掉落球；普通球命中放大奖励调用 `Ball.become_heavy()` 原地调整半径、重力、碰撞体与拖尾；重球命中放大奖励创建重球掉落球。
- `BallManager.spawn_bonus_ball()` 复用活动球注册和回收路径；成功创建时 `GameController` 才增加本局库存。回收后的定义自然进入后续批次，保证 `+1` 在本局生命周期内有效。

## 顺序发射与轨迹

首球立即从顶部缺口的中心位置发射。其余 `BallDefinition` 保留在 `BallLaunchSequence`，每经过 `ball_launch_interval_seconds` 请求下一颗预览球的入槽确认后才创建物理球，使用与首球完全相同的瞄准方向。`launch_spread_degrees` 固定为 0，禁止扇形偏转。

待发球不属于 `BallLayer`，也不参与场内物理。`Launcher` 对全部待发预览球执行局部轻量模拟：`Ball` 回收时把左右侧信息传给 `GameController`，再由 `Launcher` 赋予相反方向的内向横速度和上冲速度；预览球受重力、斜坡、低反弹推挤、速度阻尼与静止阈值约束后稳定堆叠。槽位已有球后，会使用更高的接触面阻尼；已堆叠球的微小接触只做位置修正，不重新唤醒，只有超过碰撞唤醒速度的实际撞击才恢复运动。顶部堆叠在短暂余震窗口后统一锁定，直到槽位腾空才重新参与补位，避免多球持续抖动。`BallDefinition` 同时声明普通/重球的半径、重力与伤害概率；`Launcher` 的预览和瞄准扫掠读取同一半径，避免大球视觉与碰撞不一致。两种模式均只允许一颗球穿过开口；槽位被占用后，虚线凹槽底部重新成为其余球的支撑面。后续球只有实际补到槽位才允许 `BallManager` 创建。经典模式的开口由整批回收与安全结算控制，挑战模式始终开放；实际发射时才创建 `Ball`，不同发射时刻、重力和障碍碰撞会自然形成不同轨迹。

## 物理隔离与结算

| 碰撞类别 | Layer | Mask | 结果 |
|---|---:|---:|---|
| WORLD（边界、障碍） | 1 | 0 | 提供静态碰撞体 |
| BALL（所有已发射球） | 2 | 1 | 只检测 WORLD，不碰撞其他球 |

任意球独立经过 `ACTIVE → ROLLING → LIFTING → ROOF_RETURN → recovered`。托底、生命周期超时和越界共用该状态机；兜底原因由球保留到最终 `recovered`，不会在 `ACTIVE` 状态直接注销。任意 `recovered` 只注销自身；只有待发队列和活动球集合同时为空时，`BallManager` 才发出一次 `batch_finished`，之后 Controller 才可结算并恢复 `READY`。

## 性能与测试边界

- 永久库存不设人工球数上限；当库存增长到较高数量时，应以性能剖析决定是否引入预览物理降采样或对象池，而不是丢弃奖励球。
- 回收球关闭物理推进并清除拖尾；待发球不实例化，避免无效的物理节点。
- 自动测试覆盖首帧单球、顺序队列清空、颜色隔离、球间零碰撞及全部回收后的唯一结算。
