# L5 多球并发架构

> 本文落实多球的客户端实现约束；特殊球的具体效果仍等待后续 L1 决策。
> 版本归属：v0.3 | 生命周期状态：已审校

## 设计结论

多球由 `BallBatch`、`BallLaunchSequence`、`BallManager`、每球独立的 `BallRuntimeState` 与数据化的 `BallDefinition` 协作完成。`GameController` 只编排批次和回合，`BallManager` 管理发射队列与活动球，`Ball` 只推进自身物理和回收状态。

不使用 Autoload：活动球、障碍层和回合都属于当前主场景，退出场景时应一并释放。

## 当前实现锚点

- `BallManager` 按批次维护待发队列和活动球表；两者均为空时才发出 `batch_finished(batch_id)`。
- `BallLaunchSequence` 保存待发球定义、统一瞄准方向、发射原点和发射节拍；它不是场景节点，因此不会产生等待球碰撞。
- `Launcher` 接收待发定义，并在两条顶部斜线构成的槽内按颜色绘制待发球；首球离开后队列逐颗递减。
- `BallDefinition.visual_color` 同时驱动球体和拖尾的 RGB；每个已发射球仍使用 Layer 2、Mask 1，只检测 WORLD。

## 场景职责

```text
Main
├── BallLayer
├── ObstacleLayer
├── BallManager
│   └── BallLaunchSequence（数据状态）
├── Launcher（顶部槽内的待发球预览）
└── GameController
    └── 创建 BallBatch，监听 batch_finished
```

| 模块 | 负责 | 不负责 |
|---|---|---|
| `GameController` | 创建批次、切换回合、接收批次结束、结算回合 | 逐帧遍历球、直接处理碰撞 |
| `BallManager` | 顺序创建/注册/注销球，维护待发与活动集合，发布队列视觉数据 | 伤害、得分、障碍类型判断 |
| `BallLaunchSequence` | 统一瞄准方向、待发定义、活动球表与发射节拍 | 物理移动和 HUD 绘制 |
| `Launcher` | 槽内待发球预览、瞄准输入 | 创建物理球、回合结算 |
| `Ball` | 独立运动、WORLD 碰撞、同色拖尾、报告回收 | 检测或响应其他球 |

## 顺序发射与轨迹

首球立即从顶部缺口的中心位置发射。其余 `BallDefinition` 保留在 `BallLaunchSequence`，每经过 `ball_launch_interval_seconds` 创建一颗物理球，使用与首球完全相同的瞄准方向。`launch_spread_degrees` 固定为 0，禁止扇形偏转。

待发球仅为 `Launcher` 的绘制预览，排列在顶部两条斜线形成的槽内。它们不属于 `BallLayer`，也不参与物理；实际发射后才创建 `Ball`。不同发射时刻、重力和障碍碰撞会自然形成不同轨迹。

## 物理隔离与结算

| 碰撞类别 | Layer | Mask | 结果 |
|---|---:|---:|---|
| WORLD（边界、障碍） | 1 | 0 | 提供静态碰撞体 |
| BALL（所有已发射球） | 2 | 1 | 只检测 WORLD，不碰撞其他球 |

任意球独立经过 `ACTIVE → ROLLING → LIFTING → ROOF_RETURN → recovered`。任意 `recovered` 只注销自身；只有待发队列和活动球集合同时为空时，`BallManager` 才发出一次 `batch_finished`，之后 Controller 才可结算并恢复 `READY`。

## 性能与测试边界

- 验收基线为最多 10 颗已发射球；暂不引入对象池。
- 回收球关闭物理推进并清除拖尾；待发球不实例化，避免无效的物理节点。
- 自动测试覆盖首帧单球、顺序队列清空、颜色隔离、球间零碰撞及全部回收后的唯一结算。
