# Godot 4.x 跨平台 2D 弹球消除游戏任务总控

> **创建日期**：2026-06-30  
> **目标引擎**：Godot 4.7
> **开发语言**：GDScript（全项目主逻辑统一使用；不混用 C#）
> **参考图像**：`docs/_references/微信图片_20260630224532_35_13.jpg`  
> **参考视频**：`docs/_references/游戏视频.mp4`（约 24 秒的游戏内录屏）

## 1. 任务背景

- 背景：计划使用 Godot 4.x 开发一款跨平台竖屏 2D 弹球消除游戏，玩法参考图像与参考视频中的深色界面、顶部发射位、向下弹射、数字血量几何障碍物、奖励物和回合制推进。
- 当前现状：已锁定 Godot 4.7 + GDScript，并已有核心方向和视觉参考；已完成 T1 最小闭环，后续阶段尚待实施。
- 为什么需要任务总控：该项目会跨越需求、物理、关卡生成、UI、美术、导出等多个阶段，适合拆成可独立交接的工作包，避免后续会话遗漏关键约束或越界扩张。

### 素材观测结论（总控级真值）

- 静态构图：竖屏深色蜂窝/六边形纹理背景；灰色 U 形场地边框；顶部两侧斜边留出中部开口；顶部中部为圆形发射位；底部有弧形分界与回收区。
- 游戏对象：场内出现圆形、三角形、菱形/方形、五边形等带数字血量的几何障碍；`+` 奖励物具有高对比的圆形标记；障碍以青色为主，并可使用绿色作为同一视觉体系的扩展色。
- 动态反馈：球体可同时存在多颗，画面中出现黄绿、紫等球色及黄绿彗尾式轨迹；命中会推动 Score 递增，数字障碍会随受击/清除变化。
- HUD：左上为暂停入口与 Score，右上为 Time 和双三角快捷控制图标。视频中的 Time 从 `00:08` 递增到 `00:16`，本项目 MVP 以**单局正计时**为默认语义；右上双三角控制的具体业务语义在 T4 中明确并记录。
- 证据边界：只以素材中的游戏画面作为产品真值；录屏中出现的系统级或其他应用画面不属于游戏需求。

## 7. 全局约束与注意事项

- 竖屏优先：默认设计分辨率建议从 `1080x1920` 或等比例 Canvas 基准开始，再做自适应。
- 玩法优先：任何美术细节不得破坏可读性，数字血量必须在高速碰撞和小屏幕上清晰可见。
- 参考素材使用规则：图片和视频共同构成游戏画面的真值；动态行为以视频为准、静态构图以图片为准。素材中非游戏画面一律不转化为需求。
- 物理可控：球速、反弹、最大存活时间、回合结束判定都必须可调参，不把关键参数硬编码散落在多个脚本中。
- GDScript 规范：变量、参数和返回值必须显式标注类型；节点间以类型化 signal 通信；协程只使用 `await`，不用 `yield()`；共享游戏参数使用 `.tres` Resource。
- 视觉资源策略：T1–T5 的 MVP 不新增位图美术素材；场地、球、几何障碍、HUD 与反馈优先用 `Node2D`/`Control`/主题和程序绘制实现。`assets/` 目录保留给后续图标、音效或经用户确认引入的外部资源。
- 视觉反馈克制：拖尾、受击和消除反馈必须帮助判断球路与命中，不能遮挡数字血量、危险区域或 HUD。
- 跨平台克制：避免使用只在桌面可用的输入、路径或插件能力；移动端触控是第一输入目标。
- 子任务执行严格不越界。开始某个 Tn 前先读该子任务的强制阅读，完成后只回填状态并停止。
- 过程资产只放 `_shared/`，正式项目文档放项目自身 `docs/` 目录。

- **当前状态**：已完成

#### 子任务背景

本阶段负责先把项目从“想法”变成可运行的 Godot 4.x + GDScript 工程。目标不是做完整玩法，而是建立主场景、目录结构、顶部向下发射的基础球体、一个可被击中的障碍物、基础得分和回合推进的最小闭环。

- 用户提供的玩法描述、参考图像与参考视频。

#### 要做的事情

- 创建 Godot 4.7 + GDScript 项目基础目录：`scenes/`、`scripts/`、`resources/`、`assets/`、`docs/`。

### T2 核心物理与发射系统

- **当前状态**：待完成

#### 子任务背景

弹球消除游戏的手感主要来自发射角度、多球间隔、反弹稳定性、球体回收和回合结束判定。本阶段在 T1 的最小球体基础上，把物理行为整理成可调参、可维护、可跨平台稳定运行的系统。

#### 强制阅读（精准 1-3 个）

- `docs/00-项目概览.md`：确认 T1 产出的项目结构、主场景入口和现有脚本职责。
- `scripts/game_controller.gd`：确认当前回合状态如何推进，避免物理逻辑和游戏流程互相打架。
- `scripts/ball.gd`：确认球体当前移动、碰撞、生命周期实现。

#### 输入

- T1 已完成的 Godot 项目。
- `Ball.tscn`、`ball.gd`、`game_controller.gd` 的现有实现。

#### 要做的事情

- 设计顶部向下瞄准/发射系统：支持从顶部虚线圆形瞄准位拖拽或点击确定方向，并限制接近水平的危险角度。
- 支持多球连续发射：球数、间隔、速度、起点可配置。
- 实现球体回收逻辑：触底、进入回收区、超时或离场后的统一处理。
- 建立回合结束判定：所有球回收后才允许生成下一回合。
- 增加调参入口，例如 `GameConfig` 资源或集中常量脚本；至少集中管理球数、间隔、初速度、最大反弹次数/存活时间和瞄准角限制。
- 加入基础调试显示或日志，便于验证球数、活跃球、回合状态。

#### 不做什么

- 不做障碍生成策略和道具效果（属于 T3）。
- 不做最终 UI 面板和暂停菜单（属于 T4）。
- 不做美术粒子、拖尾和高级动效（属于 T5）。

#### 预期效果

玩家能从顶部虚线瞄准/发射位连续向下发射多颗球，球体在边界和障碍之间稳定反弹，全部回收后自动进入下一回合准备状态。

#### 输出物

- `scripts/launcher.gd`：发射方向、发射队列和多球间隔控制。
- `scripts/ball.gd`：更新后的球体移动、碰撞、回收、生命周期逻辑。
- `scripts/game_config.gd` 或 `resources/game_config.tres`：核心物理参数集中配置。
- `docs/01-核心物理与发射系统.md`：参数说明、状态机、调试方法、已知限制。

#### 完成判定

- [ ] 支持至少 10 颗球按固定间隔连续发射。
- [ ] 球体不会因常规碰撞卡死在墙体、障碍或底部边界。
- [ ] 所有球回收后，回合状态准确切换为可再次发射。
- [ ] 球速、球数、发射间隔可在一个集中位置调整。
- [ ] 顶部瞄准/发射位在常见竖屏比例下可见且可操作。
- [ ] 移动端触控和桌面鼠标输入都能触发发射，且输入层保留“瞄准/发射/暂停”逻辑动作，供 T6 在 Android TV 阶段映射遥控器按键。

#### 依赖关系

- 依赖：T1
- 阻塞：T3、T4、T5、T6

#### 风险与注意事项

- Godot 物理步进与手动移动不要混用到难以预测；必须明确使用 `CharacterBody2D`、`RigidBody2D` 或自定义射线反射中的一种主策略。
- 多球数量增加后要留意节点数量和碰撞性能。

## 13. 更新规则

- 子任务开始时，将子任务总表和对应详情的状态改为 `进行中`。
- 子任务开始前，必须重新读取该子任务的强制阅读文件，不依赖上一轮会话记忆。
- 子任务完成后，将状态改为 `已完成`，立刻停止，不顺手做下一个。
- 子任务阻塞时，将状态改为 `阻塞`，并在「风险与待确认」或对应子任务下写明原因。
- 子任务产出文件后，必须回填到对应「输出物」字段；如果实际路径变化，也以回填后的路径为准。
- 发现某个一级子任务过大时，由用户显式决定是否拆成 Tn.1、Tn.2；执行者不得自行拆分总控粒度。

## 14. 过程资产放置规则

- 所有过程资产放入唯一目录 `_shared/`。
- 子任务专属资产命名为 `_shared/T{n}-{资产名}.md`，例如 `_shared/T2-物理参数试验记录.md`。
- 多文件资产使用 `_shared/T{n}-{资产名}/` 子目录。
- 任务级共享资产不加 T 前缀，例如 `_shared/玩法术语表.md`。
- 正式项目文档放入项目 `docs/` 目录，不放在总控任务目录。
- 禁止把过程资产平铺到任务根目录，禁止新建 `_T{n}/` 目录。
