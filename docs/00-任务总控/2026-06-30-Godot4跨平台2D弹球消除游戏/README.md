# Godot 4.x 跨平台 2D 弹球消除游戏任务总控

> **创建日期**：2026-06-30  
> **目标引擎**：Godot 4.x  
> **参考图像**：`docs/_references/微信图片_20260630224532_35_13.jpg`  
> **说明**：模型选择由用户在新会话开头自行决定。本总控不预定义执行模式。

## 1. 任务背景

- 背景：计划使用 Godot 4.x 开发一款跨平台竖屏 2D 弹球消除游戏，玩法参考图像中的深色界面、多球弹射、数字血量几何障碍物、回合制推进。
- 当前现状：已有核心方向和视觉参考，但尚未建立 Godot 项目结构、玩法系统、UI、美术规范、导出流水线。
- 为什么需要任务总控：该项目会跨越需求、物理、关卡生成、UI、美术、导出等多个阶段，适合拆成可独立交接的工作包，避免后续会话遗漏关键约束或越界扩张。

## 2. 总目标

- [ ] 产出一个 Godot 4.x 可运行 MVP，支持竖屏弹球消除的基础闭环。
- [ ] 建立稳定、可调参的 2D 球体物理与碰撞扣血机制。
- [ ] 建立几何障碍物、数字血量、回合生成、道具奖励的方块系统。
- [ ] 建立与参考图像一致的顶部 HUD、暂停入口、比分与时间显示。
- [ ] 建立深色背景、灰色边框、绿色/青色几何障碍、彩色球体的可维护美术还原方案。
- [ ] 建立 Android、iOS、Windows、macOS、Web 等目标平台的导出检查清单与基础配置。

## 3. 完成定义

- [ ] 项目能在 Godot 4.x 编辑器中一键运行主场景。
- [ ] 玩家能完成至少 3 个连续回合：发射多球、碰撞扣血、清除障碍、生成新障碍。
- [ ] 游戏具备 Score、Time、Pause 三类顶部 UI，并能随游戏状态更新。
- [ ] 竖屏布局在常见手机比例下不发生 UI 遮挡、出界或关键元素不可见。
- [ ] 至少完成一个桌面平台和一个移动平台的导出验证记录。
- [ ] 所有阶段输出物已回填到本总控，并标记对应子任务状态。

## 4. 范围

- 包含 Godot 4.x 项目初始化、主场景、物理球、发射流程、障碍物、回合推进、HUD、暂停菜单、基础美术资源、平台导出配置。
- 包含单机离线玩法，不依赖服务器。
- 包含面向手机竖屏优先的响应式布局，同时兼顾桌面调试窗口。
- 包含后续开发所需的脚本目录、场景目录、资源目录和文档目录约定。

## 5. 非范围

- 不做联网排行榜、账号系统、云存档、内购、广告 SDK。
- 不做复杂剧情、角色养成、关卡编辑器、UGC 分享。
- 不做商业级最终美术包、音效包、商店素材与发行宣发图。
- 不做引擎版本大迁移；默认使用 Godot 4.x 当前项目锁定版本。
- 不做性能极限优化；本总控只要求 MVP 到跨平台基础可导出。

## 6. 全局强制阅读

> 最多 1-2 个项目级文件。具体执行所需的文件放到子任务级强制阅读。

- `docs/_references/微信图片_20260630224532_35_13.jpg`：视觉参考真值，用于确认竖屏构图、深色背景、顶部信息区、球体与几何障碍的大致风格。
- `project.godot`：项目存在后必须读取，用于确认 Godot 版本、主场景、输入映射、窗口与导出基础设置。

## 7. 全局约束与注意事项

- 竖屏优先：默认设计分辨率建议从 `1080x1920` 或等比例 Canvas 基准开始，再做自适应。
- 玩法优先：任何美术细节不得破坏可读性，数字血量必须在高速碰撞和小屏幕上清晰可见。
- 物理可控：球速、反弹、最大存活时间、回合结束判定都必须可调参，不把关键参数硬编码散落在多个脚本中。
- 跨平台克制：避免使用只在桌面可用的输入、路径或插件能力；移动端触控是第一输入目标。
- 子任务执行严格不越界。开始某个 Tn 前先读该子任务的强制阅读，完成后只回填状态并停止。
- 过程资产只放 `_shared/`，正式项目文档放项目自身 `docs/` 目录。

### T1 MVP 项目骨架与基础闭环

- **当前状态**：待完成

#### 子任务背景

本阶段负责先把项目从“想法”变成可运行的 Godot 4.x 工程。目标不是做完整玩法，而是建立主场景、目录结构、基础球体、一个可被击中的障碍物、基础得分和回合推进的最小闭环。

#### 强制阅读（精准 1-3 个）

- `docs/_references/微信图片_20260630224532_35_13.jpg`：确认竖屏游戏区域、顶部 HUD、底部发射区和障碍分布的基础视觉方向。
- `project.godot`：如果项目已存在，确认主场景、Godot 版本和窗口设置；如果不存在，本阶段创建后回读确认。

#### 输入

- 用户提供的玩法描述与参考图像。
- 空项目或待初始化的 Godot 4.x 项目目录。

#### 要做的事情

- 创建 Godot 4.x 项目基础目录：`scenes/`、`scripts/`、`assets/`、`docs/`。
- 建立主场景 `Main.tscn`，包含竖屏游戏根节点、游戏区域、临时 HUD、发射点、边界碰撞体。
- 实现一个最小球体对象和一个最小数字障碍物对象。
- 实现点击或拖拽后发射一颗球，球碰撞障碍物后扣血，障碍物血量归零后消失并加分。
- 建立最小回合流程：发射、等待球回收或超时、进入下一回合。

#### 不做什么

- 不做多球队列与复杂发射角度预测（属于 T2）。
- 不做完整方块生成算法和多形状障碍系统（属于 T3）。
- 不做最终 UI 和视觉还原（属于 T4、T5）。
- 不做平台导出（属于 T6）。

#### 预期效果

项目能在编辑器中运行，玩家可以在竖屏场景里发射球、击中带数字的障碍物、看到血量减少和得分增加，并进入下一回合。

#### 输出物

- `scenes/Obstacle.tscn`、`scripts/obstacle.gd`：最小数字障碍物对象。

- `docs/00-项目概览.md`：项目结构、运行方式、MVP 当前能力说明。

#### 完成判定

- [ ] Godot 编辑器能打开项目并运行 `Main.tscn`。
- [ ] 发射球后能发生可见碰撞和反弹。
- [ ] 障碍物数字血量能减少，归零后被移除。
- [ ] Score 能在消除障碍后增加。
- [ ] 至少能连续进入 3 个回合，无脚本错误。

#### 依赖关系

- 依赖：无
- 阻塞：T2、T3、T4、T5、T6

#### 风险与注意事项

- 如果项目已经存在，严禁覆盖用户已有场景和脚本，必须先读取并适配。

### T2 核心物理与发射系统

- **当前状态**：待完成

#### 子任务背景

弹球消除游戏的手感主要来自发射角度、多球间隔、反弹稳定性、球体回收和回合结束判定。本阶段在 T1 的最小球体基础上，把物理行为整理成可调参、可维护、可跨平台稳定运行的系统。

#### 强制阅读（精准 1-3 个）

- `docs/00-项目概览.md`：确认 T1 产出的项目结构、主场景入口和现有脚本职责。
- `scripts/game_controller.gd`：确认当前回合状态如何推进，避免物理逻辑和游戏流程互相打架。
- `scripts/ball.gd`：确认球体当前移动、碰撞、生命周期实现。

#### 输入

- T1 已完成的 Godot 项目。
- `Ball.tscn`、`ball.gd`、`game_controller.gd` 的现有实现。

#### 要做的事情

- 设计发射系统：支持拖拽或点击确定方向，并限制接近水平的危险角度。
- 支持多球连续发射：球数、间隔、速度、起点可配置。
- 实现球体回收逻辑：触底、进入回收区、超时或离场后的统一处理。
- 建立回合结束判定：所有球回收后才允许生成下一回合。
- 增加调参入口，例如 `GameConfig` 资源或集中常量脚本。
- 加入基础调试显示或日志，便于验证球数、活跃球、回合状态。

#### 不做什么

- 不做障碍生成策略和道具效果（属于 T3）。
- 不做最终 UI 面板和暂停菜单（属于 T4）。
- 不做美术粒子、拖尾和高级动效（属于 T5）。

#### 预期效果

玩家能从底部发射区或指定槽位连续发射多颗球，球体在边界和障碍之间稳定反弹，全部回收后自动进入下一回合准备状态。

#### 输出物

- `scripts/launcher.gd`：发射方向、发射队列和多球间隔控制。
- `scripts/ball.gd`：更新后的球体移动、碰撞、回收、生命周期逻辑。
- `scripts/game_config.gd` 或 `resources/game_config.tres`：核心物理参数集中配置。
- `docs/01-核心物理与发射系统.md`：参数说明、状态机、调试方法、已知限制。

#### 完成判定

- [ ] 支持至少 10 颗球按固定间隔连续发射。
- [ ] 球体不会因常规碰撞卡死在墙体、障碍或底部边界。
- [ ] 所有球回收后，回合状态准确切换为可再次发射。
- [ ] 球速、球数、发射间隔可在一个集中位置调整。
- [ ] 移动端触控和桌面鼠标输入都能触发发射。

#### 依赖关系

- 依赖：T1
- 阻塞：T3、T4、T5、T6

#### 风险与注意事项

- Godot 物理步进与手动移动不要混用到难以预测；必须明确使用 `CharacterBody2D`、`RigidBody2D` 或自定义射线反射中的一种主策略。
- 多球数量增加后要留意节点数量和碰撞性能。

### T3 方块与回合生成系统

- **当前状态**：待完成

#### 子任务背景

参考图像中的核心对象是绿色/青色几何障碍物，带数字血量，分布在竖屏场地中，球碰撞后扣血并清除。本阶段负责建立障碍物类型、血量显示、回合推进生成规则和基础奖励道具。

#### 强制阅读（精准 1-3 个）

- `docs/00-项目概览.md`：确认主场景、对象目录与当前 MVP 能力。
- `docs/01-核心物理与发射系统.md`：确认球体碰撞事件、回合状态和可调用接口。
- `scenes/Obstacle.tscn`、`scripts/obstacle.gd`：确认 T1 最小障碍实现，决定扩展或重建。

#### 输入

- T2 已完成的球体碰撞与回合结束信号。
- T1 的最小障碍物对象。

#### 要做的事情

- 建立障碍物基类，支持血量、最大血量、显示数字、受击、死亡信号。
- 支持至少 4 类视觉形状：圆形、三角形、方形/菱形、五边形。
- 实现障碍生成区域和安全规则，避免直接覆盖发射点、回收区和 HUD。
- 实现回合制生成：每回合生成一批新障碍，并根据回合数提高血量或数量。
- 实现基础奖励物：例如 `+` 增加球数、低血量特殊目标或分数奖励。
- 建立失败条件：障碍到达底部警戒线或指定区域后触发 Game Over。

#### 不做什么

- 不做最终美术精修和动画包装（属于 T5）。
- 不做复杂关卡编辑器或手工关卡包。
- 不做付费道具、稀有掉落、长期成长系统。
- 不做排行榜或成就系统。

#### 预期效果

每个回合结束后，场上会按规则出现新的几何障碍物。球碰撞障碍物会扣除数字血量，血量归零时障碍消失并给分；奖励物能影响后续回合。

#### 输出物

- `scenes/obstacles/`：圆形、三角形、方形/菱形、五边形障碍场景。
- `scripts/obstacles/obstacle_base.gd`：障碍物基类。
- `scripts/obstacle_spawner.gd`：回合生成、位置选择、难度增长逻辑。
- `scripts/reward_item.gd`：基础奖励物逻辑。
- `docs/02-方块与回合生成系统.md`：障碍类型、生成规则、难度参数、失败条件说明。

#### 完成判定

- [ ] 至少 4 类几何障碍能在场景中生成并正确碰撞。
- [ ] 每个障碍显示可读数字血量，受击后数字同步减少。
- [ ] 血量归零后障碍被移除，分数增加。
- [ ] 回合推进能生成新障碍，并随回合数提升压力。
- [ ] 奖励物能被球碰撞触发，并至少支持增加后续发射球数。
- [ ] 障碍进入底部危险区域时能触发失败状态。

#### 依赖关系

- 依赖：T2
- 阻塞：T4、T5、T6

#### 风险与注意事项

- 随机生成必须可控，建议支持随机种子或调试固定种子，方便复现问题。
- 障碍数量增长要有上限，避免物理碰撞和文字渲染拖慢移动设备。

### T4 UI 与游戏状态系统

- **当前状态**：待完成

#### 子任务背景

参考图像顶部有 Pause、Score、Time，游戏需要在等待发射、发射中、回合结算、暂停、失败等状态之间清晰切换。本阶段负责正式 HUD、暂停菜单、状态管理和基础适配。

#### 强制阅读（精准 1-3 个）

- `docs/00-项目概览.md`：确认主场景结构与现有临时 HUD。
- `docs/01-核心物理与发射系统.md`：确认回合与发射状态。
- `docs/02-方块与回合生成系统.md`：确认分数、失败条件、奖励物对 UI 的影响。

#### 输入

- T1-T3 已完成的分数、时间、回合、失败状态信号。
- 参考图像中的顶部 UI 布局。

#### 要做的事情

- 建立正式 HUD：左上暂停按钮、Score 文本、Time 文本、可选球数/回合显示。
- 实现暂停菜单：继续、重新开始、退出到主菜单或关闭菜单。
- 建立统一游戏状态枚举或状态机，覆盖准备、发射中、结算、暂停、失败。
- 实现时间计数或倒计时，明确 Time 的设计含义。
- 实现 Game Over 面板，显示分数、回合数、重开按钮。
