# 设计动机：第一阶段为何这样拆分

> **理解层**：需要了解当前结构为什么适合单球原型、又为何不能直接承载下一阶段时阅读。运行时对象请查 [01-运行时对象与配置](01-运行时对象与配置.md)，完整流程请查 [02-单球回合与托底回收](02-单球回合与托底回收.md)。  
> 版本归属：V0.1-MVP | 生命周期状态：已审核

第一阶段的目标是尽快验证“顶部向下发射 → 重力反弹 → 命中数字块 → 底部回收 → 再次发射”这条完整手感闭环。因此，项目选择小场景、直接信号和一个 `GameConfig` Resource：参数能集中调节，行为可被无头测试覆盖，也无需素材或复杂关卡数据。

`Main` 只组合并连接 UI；`GameController` 只裁决回合与得分；`Ball` 只承担当前一颗球的运动；`Obstacle` 只承担当前一块六边形的生命。这个边界让 T1 可读、可验证，但它是有意为之的“单实例原型”，不是未来内容系统的最终模型。

底部托底不是物理碰撞体。活动球检测到共享曲线后，切换到受控的回收状态，依次滚出、上行和回顶。这使路线稳定、视觉和逻辑共用同一个曲线公式；代价是该路线目前写在 `Ball` 内，不会自动适配不同球半径或变形场地。

下一阶段的关键不是马上加“爆炸球”分支，而是先把“静态定义、运行时实例、命中结算、场地路线、回合编排”拆开。这样普通球仍能复用现在的链路，特殊球和特殊障碍只是额外的组合，而非改写核心脚本。
