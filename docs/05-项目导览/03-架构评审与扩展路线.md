# 架构评审与扩展路线

> **设计建议（未实施、未作为玩法需求拍板）**：计划多球、波次下压、能力球或多类障碍前阅读。当前实现事实见 [01-运行时对象与配置](01-运行时对象与配置.md)。  
> 版本归属：V0.2（建议） | 生命周期状态：草案

## 结论

第一阶段架构对“单球 + 单个固定六边形障碍”是合理的：职责小、配置集中、信号简单、关键路径已有无头回归。它**不适合直接叠加**多球、特殊球和丰富障碍；这些能力会迅速让 `Ball`、`Obstacle`、`GameController` 相互识别具体类型。

建议在实现第一个特殊球前，先完成“实体定义资源 + 命中协议 + 活动实体集合 + 场地回收服务”的最小加固。无需立刻做大型框架或 Autoload。

## 主要风险与优先级

| 优先级 | 现状 | 风险 | 建议 |
|---|---|---|---|
| P0 | Controller 只有 `_active_ball` / `_active_obstacle` | 任一球回收即结束回合，无法支持分裂、多球齐落或多块关卡。 | 引入 `BallManager`/活动球集合；仅集合清空才结算回合。关卡障碍交给 `LevelSpawner` 管理。 |
| P0 | Ball 直接识别 `Obstacle` 并 `take_hit()` | 爆炸、穿透、暴击和材质反弹会产生类型分支。 | 定义 `HitContext` 与统一 `receive_hit(context)`；Ball 只报告碰撞，结算器决定伤害和反弹。 |
| P0 | 球的物理碰撞半径固定 20 | 大球会出现“看起来很大、碰撞仍很小”的错误。 | 为每个球实例同步 `CircleShape2D.radius`、绘制、拖尾、托底偏移和回收安全距离。 |
| P1 | Obstacle 把六边形形状、绘制、生命、得分写在一起 | 新增三角、圆、移动、弹簧、炸弹块时易造成视觉/碰撞不一致。 | 用 `ObstacleDefinition` 描述形状、HP、得分、材质；用独立场景/行为组件实现静态、移动、触发等行为。 |
| P1 | 回收路线在 Ball 内 | 场地变化、巨型球或飞行球会重复实现路线。 | 提取 `ArenaGeometry` / `BallRecoveryPath`，从球半径计算安全航点。 |
| P1 | ACTIVE 每帧只处理一次碰撞 | 高速球、小障碍、穿透连续命中可能漏碰撞。 | 设计有限次数的剩余位移迭代或子步策略，并为高速场景做回归。 |
| P2 | 每次直接 instantiate/queue_free | 多球、爆炸、特效时产生分配尖峰。 | 在同时实体数量稳定后加入球和特效对象池；回收对象禁用物理处理。 |
| P2 | 输入只支持鼠标/触摸 | Android TV 无法直接复用。 | 后续将 `Launcher` 拆为输入命令与视图；通过 Input Map 接入方向、确认、返回，并注意 Godot 4.6+ 的双焦点行为。 |

## 建议的目标组合

```text
BallDefinition(Resource) ─┐
BallEffect(Resource/组件) ├→ Ball（仅移动、生命周期与实例状态）
                          └→ HitResolver（伤害、反弹、能力结算）

ObstacleDefinition(Resource) + ObstacleBehavior/场景
  → Obstacle（统一 receive_hit）

WaveDefinition/SpawnRule → LevelSpawner
BallManager（活动球集合） → GameController（只做回合/分数编排）
ArenaGeometry/RecoveryPath → 所有需要回收的球
```

Resource 是静态定义，不在运行时写回。若某局需要变体，复制为实例运行状态；Godot 4.5+ 对有嵌套 Resource 的定义可使用 `duplicate_deep()`。

## 三类球的落地方式

| 类型 | 最小定义字段 | 命中规则 | 额外约束 |
|---|---|---|---|
| 爆炸球 | `explosion_radius`、`damage`、`trigger` | 首次有效命中或寿命结束时，范围查询目标并逐个发送 `HitContext`。 | 定义是否连锁、同一目标去重、爆炸后是否立刻回收。 |
| 穿透球 | `pierce_count`、`damage_decay` | 命中后不反射或按规则衰减；以实例 ID 集合防止同一障碍重复扣血。 | 需要处理一帧内连续命中与剩余位移。 |
| 大体积球 | `radius`、`gravity_scale`、`speed_scale` | 使用同一命中协议。 | 同步碰撞形状、生成净空、托底高度、墙外与屋顶回收路径。 |

## 障碍和反弹的扩展原则

- 几何形状与玩法行为分离：圆形、三角、六边形是几何；耐久、移动、弹簧、分裂、护盾是行为。
- 反弹不应再由球读取全局单一系数。建议每个障碍定义 `BounceProfile`（保速比例、角度修正、是否吸附/穿透）。
- 所有可受击对象使用同一入口，避免 `if collider is X` 链；范围攻击通过物理空间查询或可受击对象注册表筛选。
- 多球回合必须定义“所有球回收才结束”、同帧多次摧毁的得分顺序、爆炸与穿透的去重规则。

## 测试与性能清单

- 每种球 × 每种障碍：命中、反弹、伤害、得分和销毁。
- 多球同时回收、空场、同帧多障碍销毁、分裂后回合只结算一次。
- 爆炸半径边界、穿透次数与同目标去重、巨球窄通道与托底路径。
- 高速/低帧率下的墙、斜顶、小障碍和连续命中。
- 多球 + 多障碍 + 特效的帧时间、对象池回收、暂停恢复。
- Android 真机通过后，再补 Android TV 的焦点、方向键、确认/返回冒烟测试。
